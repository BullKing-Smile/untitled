### Sentinel 
#### Sentinel 的 Warm Up的时间设置 依据什么来定？
预热时间 = 系统达到稳定状态所需的最长时间 + 安全缓冲
系统达到稳定状态所需的最长时间 ，这个要算**JVM 预热耗时、缓存加载时间、线程池就绪时间、依赖服务就绪**。
**<font color=orange>简单的说，至少30秒，推荐45秒</font>**。

#### 流量突增的情况，低负载到QPS阈值 sentinel是如何判断的低负载的？
通过 **滑动窗口** （比如默认1秒拆分为20个格子）实时计算 QPS
当检测到 当前QPS < 冷启动阈值 时，自动启用预热模式
冷启动阈值 = 稳定阈值 / coldFactor
coldFactor ：默认 3（即初始阈值仅为稳定 QPS 的 1/3）











### **电商系统设计（QPS 1万）**

- **Nginx集群**：3台（每台处理3000 QPS，负载均衡）
- **Spring Cloud Gateway**：2台（限流阈值5000 QPS/台）
- **商品服务**：4个实例（Tomcat配置500并发/实例，共2000并发）
- **Redis集群**：6节点（3主3从，支撑10万QPS读请求）
- **MySQL**：1主2从（读请求走从库）



#### **(1) Nginx**

- **理论值**：单机可支持 **50万~100万** 并发连接

- **实际值**：受限于CPU、内存、网络带宽，通常能处理 **2万~5万** QPS

#### **(2) Spring Cloud Gateway**

- **理论值**：单实例可处理 **1万~3万** QPS
- **实际值**：通常 **3000~8000** QPS

#### **(3) Tomcat**

- **NIO模式**（默认）：**2000~5000** 并发（受线程池配置影响）。

> 🔥 **对比总结**：
>
> - **静态请求**：Nginx > Spring Cloud Gateway > Tomcat
> - **动态请求**：Tomcat是最终处理者，性能瓶颈通常在此，需横向扩展。
